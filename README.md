# Blackmagic Control Pro

An iPad app that turns a USB video feed plus Bluetooth into a monitor-and-control station for Blackmagic cameras. Plug the camera's video into the iPad, pair over Bluetooth, and you get a full-screen live image with camera controls layered on top — focus, iris, shutter, ISO, white balance, ND, audio levels, record start/stop — without touching the camera body.

This is an unofficial hobby project. It is not made by, endorsed by, or affiliated with Blackmagic Design.

## How it works

There are two independent paths between the iPad and the camera:

- **Video** comes in over USB-C as a UVC capture device — either the camera's own webcam output (via a hub) or an HDMI-to-USB capture card. iPadOS 17 exposes external cameras through AVFoundation, and the app renders that feed full-screen.
- **Control** goes over Bluetooth LE using Blackmagic's published Camera Control Protocol (the CCU packet format documented in `BlackmagicCameraControl.pdf`, included in the repo). The app encodes CCU packets directly — there's no companion box or REST bridge involved — and decodes the camera's outgoing state stream so the UI reflects changes made on the camera itself, not just ones made in the app.

Requires an iPad with USB-C on iPadOS 17+, and a Bluetooth-capable Blackmagic camera on firmware 8.6 or newer. Developed and tested primarily against a BMPCC.

## What it can control

Everything the CCU protocol exposes that made sense for a monitor app: lens (focus, one-push AF, iris by stop/normalised/ordinal, zoom, OIS), exposure (shutter angle or speed, ISO, gain, auto-exposure modes), color (white balance + tint, auto WB), recording format and codec, dynamic range, sharpening, ND filter, display LUTs, audio (input type and levels, phantom power, headphone/speaker mix), transport (record/stop, media slot status), and the camera's own monitor overlays (frame guides, safe areas, zebra, focus peaking, display brightness).

The iPad-side HUD adds its own frame guide and safe-area overlays on the live image, independent of what the camera is drawing.

## Repo layout

```
BlackmagicControl/
  BLE/            CCU packet encoder/decoder, CoreBluetooth controller,
                  connection state machine (scan → pair → connect → reconnect)
  Domain/         Normalized camera state model decoded from the CCU stream
  Preview/        External (UVC) camera capture and preview rendering
  UI/             Monitor HUD: top/bottom bars, parameter panels, pairing,
                  settings, frame guide overlay
  Diagnostics/    On-device logging, crash capture, redaction, and a
                  one-tap "email me the diagnostics zip" exporter
BlackmagicControlTests/   Unit tests, heaviest around packet encoding/decoding
docs/             Tester guide (with a PDF generator) and a hardware
                  test checklist for on-camera verification
scripts/          release.sh builds a versioned unsigned IPA + dSYMs into dist/
```

The Xcode project is generated — edit `project.yml`, not the `.xcodeproj`.

## Building

You need Xcode 16+ and [XcodeGen](https://github.com/yonaskolb/XcodeGen):

```sh
xcodegen generate
open BlackmagicControl.xcodeproj
```

Run the `BlackmagicControl` scheme on an iPad (or simulator — the UI works, but obviously no BLE camera or UVC feed there). Tests:

```sh
xcodebuild test -project BlackmagicControl.xcodeproj -scheme BlackmagicControl \
  -destination 'platform=iOS Simulator,name=iPad Pro 13-inch (M4)'
```

Note that most of the BLE and preview code can only be truly verified against real hardware; `docs/hardware-test-checklist.md` is the script for that.

## Releasing to testers

There's no App Store presence. Builds are unsigned IPAs that testers sideload:

```sh
scripts/release.sh
```

This stamps the build number, archives, and drops the IPA, xcarchive, dSYMs, and the source commit hash into `dist/v<version>-<build>/`. Testers follow `docs/tester-guide.md`, which walks through sideloading with iLoader step by step (there's a PDF version generated by `scripts/generate_tester_guide_pdf.py` for people who won't read markdown).

## Status

Alpha. The core loop — pair, monitor, control, record — works. Rough edges are expected; the About tab has a Send Diagnostics button that emails a redacted log bundle, and that's the preferred way to report problems.
